# DeviceLink 项目架构图

## 1. 整体分层架构

DeviceLink 基于 OSI 七层模型简化为五层架构，每层职责清晰，模块解耦。

## 3. 各模块详细职责

### 3.1 DeviceLink.Transport（物理传输层）

| 职责 | 说明 |
|------|------|
| 字节流传输 | 提供原始字节的读写能力 |
| 物理连接管理 | 打开/关闭物理连接 |
| 电气信号控制 | DTR、RTS等信号控制 |
| 缓冲区管理 | 接收/发送缓冲区管理 |

**实现类**：

| 类名 | 适用场景 | 通信方式 |
|------|----------|----------|
| `SerialPortTransport` | RS232/RS485设备 | 串口通信 |
| `TcpTransport` | 网络设备 | TCP/IP |
| `UdpTransport` | 无连接网络设备 | UDP |
| `UsbTransport` | USB设备 | USB |
| `LoopbackTransport` | 单元测试 | 回环 |

### 3.2 DeviceLink.DataLink（数据链路层）

| 职责 | 说明 |
|------|------|
| 帧封装 | 将数据封装为协议帧 |
| 帧解析 | 从字节流中识别帧边界 |
| CRC校验 | 计算和验证CRC校验码 |
| 错误检测 | 检测传输错误 |

**帧策略实现类**：

| 类名 | 帧格式 | 适用协议 |
|------|--------|----------|
| `DelimiterFrameStrategy` | [数据][分隔符] | ConST(\0)、SCPI(\n)、AT(\r\n) |
| `FixedLengthFrameStrategy` | [定长数据] | 自定义定长协议 |
| `ModbusRtuFrameStrategy` | [数据][CRC16] | Modbus RTU |

### 3.3 DeviceLink.Protocol（协议层/表示层）

| 职责 | 说明 |
|------|------|
| 命令编码 | 将业务命令编码为协议格式 |
| 响应解码 | 将协议响应解码为业务数据 |
| 错误检测 | 识别设备错误响应 |
| 格式转换 | 数据格式转换（如字节序） |

**编解码器实现类**：

| 类名 | 协议 | 帧格式 |
|------|------|--------|
| `ConSTCodec` | ConST | `地址:R/W:命令:参数:\0` |
| `ScpiCodec` | SCPI | `COMMAND?\n` |
| `ModbusRtuCodec` | Modbus RTU | `[从站][功能码][数据...]`（不含CRC） |

### 3.4 DeviceLink.Session（会话层）

| 职责 | 说明 |
|------|------|
| 连接管理 | 管理连接生命周期 |
| 请求-响应 | 实现请求-响应模式 |
| 超时重试 | 处理超时和重试逻辑 |
| 并发控制 | 线程安全的并发访问 |

**会话实现类**：

| 类名 | 适用场景 | 特点 |
|------|----------|------|
| `DirectSession` | 串口/TCP直连 | 通过DataLink收发 |
| `MqttSession` | MQTT物联网 | 通过MQTT Broker通信 |
| `LoopbackSession` | 单元测试 | 回环测试 |

### 3.5 DeviceLink.DeviceBase（设备基类库）

| 职责 | 说明 |
|------|------|
| 命令发送 | 提供SendAsync/SendForResultAsync |
| 错误检查 | 自动检查设备错误响应 |
| 日志记录 | 记录通信日志 |
| 管道组装 | 组装完整的OSI通信栈 |

### 3.6 DeviceLink.Pipeline（管道构建器）

| 职责 | 说明 |
|------|------|
| 管道构建 | 通过Builder模式组装通信栈 |
| 依赖注入 | 注入各层实现 |
| 配置管理 | 管理通信配置 |

## 4. 数据流转过程

### 4.1 发送数据流程

```
应用层: UMC1200.GetTemperaturePVAsync()
    │
    ▼ Command.Read("3.0.1")
协议层: ModbusRtuCodec.Encode()
    │
    ▼ [01][03][00][00][00][01] (原始数据，不含CRC)
会话层: DirectSession.SendAndReceiveAsync()
    │
    ▼ 管理超时重试
数据链路层: ModbusRtuFrameStrategy.BuildFrame()
    │
    ▼ [01][03][00][00][00][01][84][0A] (添加CRC16)
物理层: SerialPortTransport.WriteAsync()
    │
    ▼ 字节流
设备: UMC1200温湿度控制器
```

### 4.2 接收数据流程

```
设备: UMC1200响应
    │
    ▼ 字节流
物理层: SerialPortTransport.ReadAsync()
    │
    ▼ [01][03][02][00][C8][B9][F6]
数据链路层: ModbusRtuFrameStrategy.TryParseFrame()
    │
    ▼ [01][03][02][00][C8] (剥离CRC，验证校验)
会话层: DirectSession
    │
    ▼ 返回响应字节
协议层: ModbusRtuCodec.ExtractRegisters()
    │
    ▼ [200] (寄存器值)
应用层: UMC1200.GetTemperaturePVAsync()
    │
    ▼ 200 (20.0℃)
```

## 5. 模块职责划分原则

| 层级 | 职责 | 不该做 |
|------|------|--------|
| **应用层** | 业务逻辑、设备特定操作 | 协议细节、帧格式 |
| **会话层** | 连接管理、超时重试 | 数据编解码、帧处理 |
| **协议层** | 数据编解码、格式转换 | CRC校验、帧边界检测 |
| **数据链路层** | 帧封装、CRC校验、帧同步 | 业务逻辑、连接管理 |
| **物理层** | 字节流传输、电气信号 | 任何协议处理 |

## 7. 扩展新设备指南

### 7.1 创建新设备类

```csharp
public class MyDevice : DeviceBase
{
    // 1. 选择合适的构造函数
    public MyDevice(string portName, byte slaveAddress = 1)
        : base(portName, new MyProtocolCodec(slaveAddress), new MyFrameStrategy())
    {
    }

    // 2. 实现业务方法
    public async Task<int> GetDataAsync(CancellationToken ct = default)
    {
        return await SendForResultAsync(
            Command.Read("GET_DATA"),
            raw => ParseResponse(raw),
            ct);
    }
}
```

### 7.2 选择帧策略

| 协议特征 | 推荐帧策略 |
|----------|------------|
| 以特定字节结尾（如\0、\n） | `DelimiterFrameStrategy` |
| 固定长度帧 | `FixedLengthFrameStrategy` |
| 带CRC校验的二进制协议 | `ModbusRtuFrameStrategy` |

### 7.3 选择会话类型

| 通信方式 | 推荐会话类型 |
|----------|--------------|
| 串口/TCP直连 | `DirectSession` |
| MQTT物联网 | `MqttSession` |
| 单元测试 | `LoopbackSession` |

## 10. 总结

DeviceLink 采用清晰的五层架构设计：

1. **物理层**：负责原始字节传输
2. **数据链路层**：负责帧封装和CRC校验
3. **协议层**：负责数据编解码
4. **会话层**：负责连接管理和超时重试
5. **应用层**：负责业务逻辑

每层职责单一，通过接口解耦，支持灵活扩展和替换。
